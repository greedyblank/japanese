# 0001. 标签云：从「词性平铺」迁移到「《分類語彙表》三层语义树」

- 日期：2026-06-22   状态：已采纳

## 背景

`kana_etymology_map.html` 从韩语 `etymology_map.html` fork 而来，**标签云的渲染层**（卡片网格 + inline 展开 + 全屏树浏览器）**早已完整移植**，但标签云实际显示的是一堆平铺的**词性卡片**（名词 86 / 动词 77 / 一段 17…），12 桶语义分类从未生效。

根因是一个**未完成的数据迁移**：
- spec（`docs/superpowers/specs/2026-06-22-ninjal-entry-tags-design.md`）第 13 行要求「词性、活用不能写入 tags；每词一个《分類語彙表》12 桶语义标签」。
- 但 192 条种子的 `tags` 全是词性，`pos` 字段一个没填，`tagGraph` 是空 `{}`。
- 于是 `getRootTags()` 把所有词性标签当成「孤儿根」平铺，12 桶根本不出现。

韩语能分层是因为 `tags` 是**纯语义**（烹饪/气象/身体变化）+ 词性在独立的 `pos` 字段；日语把词性塞进 `tags` 导致标签云退化为词性平铺。

## 抉择

1. **树深度**：两层（桶→词条）vs 三层（桶→子类→词条）。三层更贴合「树状分层级」诉求，且现有渲染层（`expandedCardHtml` 子标签 pill 流 + `renderTagTree` 缩进树）可直接复用；两层则需为「根下无子类」补渲染分支。
2. **语义标签来源**：离线映射表（基于《分類語彙表》逐词分配，写入种子）vs 运行时 AI 生成。离线映射 deterministic、可审查、不依赖 API key，符合 spec「离线重建、不调 LLM」。

## 决定

- **三层树**：12 桶（《分類語彙表》体/用/相/その他）→ **35 个语义子类** → 192 词条。子类即 `entry.tags` 的值，桶由 `tagGraph` 的 parent 链推导。
- **离线映射**：种子 `ENTRIES` 的 `tags` 直接写为子类名；词性迁移到 `pos` / `conjugation` / `wordOrigin` 字段。
- **tagGraph 构建**：新增 `TAG_TAXONOMY`（桶→子类数组）+ `buildDefaultTagGraph()`，在 `init()` 里 tagGraph 为空时注入。
- **run-time 迁移** `migrateLegacyTagsToGrammar()`：种子词条（default-N）若 tags 仍是旧格式则同步最新种子分类；用户新增词条把词性/活用 tag 迁到 grammar 字段。幂等，复刻 `migrateDefaultKanaReadings` 模式。

## 后果

- 标签云显示 12 桶卡片，点桶展开看子类 pill，点子类看词条。
- 词性以 `pos` badge 展示（`posBadgeClass` / `renderDetail` 已就绪），不再污染标签云。
- 35 个子类中部分只有 1 词（飲食物 / 文化·芸術 / 天象·気象 等），保留因语义精确、可随词条增长自然充实。
- spec 原描述（扁平 12 桶、每词一个根标签）被**扩展为三层**；spec 验收「192 词各有一个标签」仍满足（每词恰好一个子类，子类属于桶）。
- 韩日差异处理：日语三种语源（和語/漢語/外来語）异质，不能用韩语式的单音根做语义分类，故采用跨语种通用的《分類語彙表》语义桶，而非照搬韩语 20 大类或音核聚合。
