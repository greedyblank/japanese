# 0002. 多端同步：GitHub Gist + id 合并 + tombstone

- 日期：2026-06-26　状态：已采纳

## 背景

`kana_etymology_map.html` 是单文件前端应用，所有数据存浏览器 `localStorage`，各端独立、互不知情。需求：在 iOS 手机等多端「加词即同步」。

现状约束：

- 数据：`localStorage`（`japanese_kana_map_v1` / `japanese_kana_taggraph_v1` …），已有 JSON 导入/导出
- AI 补全：走本地 Python 代理 `glm_proxy.py`（`localhost:8787`），**手机端连不上**
- File System Access API（`showSaveFilePicker`）：**iOS Safari 不支持**
- 用户是个人开发者，已有 git / GitHub 工作流

## 抉择

**载体（iOS 怎么跑）**

- Capacitor 原生打包 + Apple 签名（$99/年）：只为桌面图标过度工程 ❌
- ✅ PWA：`manifest.json` + service worker，Safari「添加到主屏」即可

**同步后端**

- WebDAV / 坚果云：要赌浏览器 CORS 支持 ❌
- BaaS（Supabase / Firebase）：要改数据层、绑死供应商 ❌
- 自建 Cloudflare Worker：对个人工具过度 ❌
- ✅ GitHub Gist：纯前端 `fetch` 读写**私有** gist，零后端代码、零运维、复用现有 GitHub 账号

**合并策略**（防止两端并发互相覆盖丢词）

- 全量快照覆盖：两端同时操作必丢词 ❌
- 全量 CRDT：过重 ❌
- ✅ entry 级 last-write-wins：每个 entry 已有**全局唯一 id**（`user-<ts>-<rand>` / `default-N`），加 `updatedAt` 时间戳，按 id 取并集、`updatedAt` 大者胜。**关键：稳定 id 本就具备**，合并最难的一关已过

**删除如何同步**（全量覆盖模型的固有坑）

- 内联软删除（entry 留 `deletedAt` + 改所有渲染 filter）：侵入大 ❌
- ✅ 独立 tombstone 数组 `state.tombstones`：删除移到墓碑，**渲染层零改动**；合并时 `deletedAt >= updatedAt` 的 entry 被剔除；删除后又编辑（`updatedAt > deletedAt`）自然复活

**tagGraph 合并**

- 给每个 tag 节点加 updatedAt：要改所有 tagGraph 写入点，侵入大 ❌
- ✅ 整体 last-write-wins（`tagGraphUpdatedAt`，内容签名检测到变化才盖戳）：tag 改动低频，corner case 概率极低且后果轻

**即时性 / 防环**

- WebSocket 真毫秒推送：加词低频，不值 ❌
- ✅ 上传 debounce 1.5s（改了才传）；下载只在打开/刷新/切回前台时触发（**不做定时轮询**）。用户不需要"不刷新也即时"，去掉轮询后 API 调用次数大降，避免撞 GitHub secondary rate limit
- ✅ payload 内容签名比对未变则跳过 push / 合并，杜绝 save→push→pull→save 循环

## 决定

新增 `CLOUD SYNC` 模块（纯前端，GitHub Gist）：

- `mergeState(local, remote)` 纯函数：entries 按 id+updatedAt 并集；tombstones 按 id+deletedAt 并集并应用；tagGraph 整体 LWW
- `pushToGist` / `pullFromGist`：`fetch` GitHub API，内容签名防环
- `scheduleCloudSync`：挂在 `saveToStorage` 末尾，debounce 上传
- `startCloudSync`：启动 + `visibilitychange`（切回前台）触发拉取；`_visibilityBound` 防重复绑 listener。**无定时轮询**
- UI：☁️ 按钮 + sync-modal（Token / Gist ID / 自动同步 / 状态），`createGist` 一键建私有 gist

数据地基改动：

- 每个 entry 打 `updatedAt`（`saveEntry` / `bulkImport` / `applyAIEntry`）
- `state.tombstones`；`deleteCurrent` 改为写墓碑
- `tagGraphUpdatedAt`，`saveToStorage` 用 tagGraph 内容签名决定是否盖戳
- `exportJSON` / `importJSON` 带上 tombstones，保证导入导出也传递删除记录

PWA：`manifest.json` + `sw.js`（HTML network-first / 静态 cache-first）+ PIL 生成的 PNG 图标。

## 后果

- **新增依赖**：无（纯前端 + GitHub REST API）。需用户自备 GitHub Token（gist 权限）。
- **数据迁移**：旧 entry 无 `updatedAt`（视为 0），首次同步以远端为准，行为安全。
- **已知限制**：
  1. tagGraph 整体 LWW：两端离线各自改 tag 后重连，可能丢一方的 tag 编辑（entry 不受影响）。可接受。
  2. tombstone 无限累积：长期使用墓碑数组会增长（可后续加定期清理，如保留 90 天）。
  3. AI 补全仍需代理：`glm_proxy.py` 手机端不可用；若 GLM 浏览器直连不通，需另部署 Cloudflare Worker（**未含**在本决策，见后续）。
  4. GitHub gist secondary rate limit：短时间大量建/改/删 gist 会触发 403（曾因测试反复建删触发）。已去掉定时轮询 + 撞限流自动冷却 5 分钟缓解；日常（仅刷新拉取 + 改了才传）调用极低，基本不撞。
- **部署**：GitHub Pages（静态托管），命令见《多端同步使用指南》。
- **后续可选**：Cloudflare Worker 替代 glm_proxy；tombstone GC；tagGraph 节点级时间戳（若 tag 编辑变频繁）。
